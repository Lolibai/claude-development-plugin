## Always-apply invariants

Non-negotiable. These fire on every change without loading a skill. Items tagged *(gated)* apply
only when `.claude/stack.md` declares that capability — otherwise they're a no-op.

1. **Sensitive-data protection** *(gated on `compliance` ≠ none — e.g. HIPAA, GDPR, PCI)* —
   Treat regulated/sensitive data as encrypted at rest and audited on mutation per the project's
   regime. Never log or return sensitive data in errors. Even when `compliance` is `none`, never
   leak secrets or PII.
2. **Respect architectural boundaries** — Follow the dependency direction and access conventions
   documented in `.claude/stack.md` and the project's architecture docs (common examples:
   inward-only layering `domain ← application ← infrastructure ← presentation`; services reached
   through an injected context rather than imported directly; lazy infra init, not at module
   load). Don't cross a boundary the project forbids.
3. **UTC dates** — All dates are UTC at rest, in transit, and in comparisons. No local time.
4. **No magic literals for domain concepts** — Use enums/constants (validated at boundaries)
   instead of hardcoded strings. Don't redeclare shared option lists or constants — import the
   project's canonical ones.
5. **Language strictness** *(gated on TypeScript)* — `??` for null/undefined defaults, `||` only
   for true falsy coercion; no non-null `!` assertions (use type guards); don't mark functions
   `async` without `await`; prefer `satisfies` / `as const` / type guards over `as` casts; prefix
   intentionally-unused params with `_`. For other languages, apply the equivalent strict-mode
   lint rules the project enforces.
6. **Generic, non-enumerating errors** — User-facing messages don't reveal which factor failed
   (no "user not found" vs "wrong password"). Log detail internally; surface appropriate codes.
7. **Commit format** — Follow `commands.commitConvention`. When the issue tracker defines a
   `keyPrefix`, lead the subject with the `<KEY>` (e.g. `<KEY> type(scope): description`).
   Lowercase imperative subject, no trailing period; split work that spans scopes into separate
   commits. If `commands.commitNoAttribution` is set, **no assistant attribution.**
8. **Verify after substantive changes** — Run the project's `commands.typecheck`, `commands.lint`,
   and targeted tests before claiming done. Never assert success without running them.
9. **LEGO philosophy — component-first UI (smart/dumb split)** *(frontend projects)* — Build UI
   from named components, not anonymous `<div>` trees: **before writing `<div className="...">`,
   check whether a component already exists**; if it does, use it; if not, name and extract it; if
   the same markup appears twice, extract it. The full rule (dumb vs smart layers, the per-project
   component inventory, the red-flags list) lives in the **`lego-philosophy`** skill — invoke it
   when building or editing any UI. This file states the rule; the skill carries the detail.
10. **Viewport-fit layouts — `dvh`, never `vh`** *(frontend projects)* — Any fill-the-screen /
    above-the-fold region must size to the **dynamic** viewport so it fits at every resolution.
    - Use `min-h-dvh` / `100dvh`, never `min-h-screen` / `100vh` (`vh` is the *large* viewport and
      ignores browser chrome and mobile toolbars, so content spills past the fold).
    - Use `min-h-*`, never `h-*`, for content regions (`min-h-dvh` grows when content is taller;
      `h-dvh` clips).
    - **The document scrolls — never lock it.** No `body{overflow:hidden}` or inner
      `h-screen overflow-y-auto` scroller that traps content out of normal page scroll and
      full-page screenshots.
    - Sticky-footer structure: a `min-h-dvh flex flex-col` region holds header + main; the footer
      sits after it.
    - Decorative/hero `<img>` out of flow: `absolute inset-0 h-full w-full object-cover` inside a
      `relative` box (an in-flow `h-full w-full` img inflates the column when the parent has no
      definite height).
    - Verify across resolutions (≥ 1512×982, 1920×1080, 2560×1440); E2E assertions relative to
      `window.innerHeight`, never a hardcoded pixel height.
    *(Adapt the class names to `frontend.styling`; the principle holds for any CSS approach.)*
11. **AC gate — re-verify acceptance criteria before every PR (mandatory)** — Every implementation
    flow ends with an explicit AC double-check before the PR is opened or updated.
    - **Source of truth:** the issue tracker's AC (from `issueTracker` in stack.md) **plus every
      linked design/spec doc** (e.g. wiki/Confluence pages, `design.figma` frames), harvested
      fresh — not from memory. If the branch carries a `<KEY>`, fetch that issue; if AC was given
      inline, use it verbatim. **If linked spec docs exist, reading them is mandatory** — "the
      ticket description was enough" is not a valid skip. On a fetch failure, report it and get
      explicit sign-off before proceeding.
    - **The check:** for each AC item, state where it's implemented (file/component/procedure) and
      which test covers it. An item with no implementation pointer or no covering test is a
      **gap** — resolve it (implement, cover, or get explicit sign-off to defer with a ticket
      reference) before proceeding.
    - **The artifact:** the PR body contains an `## ACs covered` section — one row per item:
      criterion → implementation pointer → covering test → status
      (`covered | deferred (<KEY>) | n/a (reason)`). A ticket with genuinely no AC says so
      explicitly; silence is not allowed.
    - Skills **invoke** this gate; they must not restate it.
12. **Lesson loop — the knowledge store is the single source of truth (mandatory)**
    *(gated on `memory.store` ≠ none)* — Lessons are never hardcoded into skills or this file; they
    live only in the configured memory store.
    - **Ask (read) — every implementation task.** Query the memory store (from `memory.store`) for
      lessons relevant to the task (keyed by `<KEY>`, feature/bug keywords, touched files/domain)
      and apply each as a binding constraint.
    - **Store (write) — after every bug resolution.** Save the lesson (root cause, wrong-vs-right
      behavior, the guard that prevents recurrence) to the memory store. Update an existing entry
      instead of duplicating. The fix isn't done until this is stored.
    - Skills carry only the *instruction* to query/store — the lesson content stays in the store so
      there is one place to update it.

## Skills map — route by intent

Claude Code auto-surfaces skills via their `description:`. Use this map when several fit or to
pre-plan a session. Skills read `.claude/stack.md`; anything the config marks `none` is skipped.

| Intent | Skill(s) |
|---|---|
| Bug fix / ticket / Gherkin scenario from the tracker | `devfix` |
| New backend feature / endpoint / use case | `backend-feature-workflow` |
| DB schema change / migration | `database-migration` |
| Edit a serverless / edge function | `serverless-function` |
| Build or edit any UI (component-first / smart-dumb split) | `lego-philosophy`, `frontend-component-conventions` |
| Design → code | `figma-plan-and-validate` → `implement-designs` |
| Mobile responsiveness audit | `mobile-friendly-checker` |
| Scaffold a test harness from scratch (E2E + unit, Playwright + Gherkin, page objects / hooks) | `scaffold-test-projects` |
| E2E (BDD / `.feature`) fixing or coverage | `e2e-narrow-fail-focus-success` |
| Test-management ↔ E2E mirrors | `test-management-sync` |
| Verify / double-check after implementation | `double-check-code`, `generate-tests-after-implementation` |
| Run targeted test suites | `run-tests` |
| Store / retrieve decisions, incidents, rules | `memory-first`, `memory-validator` |
| Review a GitHub PR | `github-pr-review` |
| Architecture / compliance / dependency-direction review | `principal-architect` |
| Set up or update the project config | `onboard` |
| Launch / stop the autonomous loop stack | `launch-loop-stack`, `stop-loop-stack` |

(Adjust to the skills actually installed; the loop stack ships generic, config-driven versions of
each tool-specific skill.)
